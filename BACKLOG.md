## High — correctness, required for first release

| Title | Scope |
|---|---|
| `setattr` is unimplemented — chmod/chown/utimes and especially **truncate/ftruncate** | The co-daemon implements no FUSE `setattr` op (no `fn setattr` in `mount/src/fuse_fs.rs`). Two consequences: (1) `chmod`/`chown`/`touch -d` (utimes) return ENOSYS — cosmetic for a single-user cloud mount; (2) the load-bearing one — **truncate/ftruncate (a SETATTR carrying `FATTR_SIZE`) is unhandled**, so `>` overwrite of an existing file (`open(O_TRUNC)` → kernel issues SETATTR size=0), the `truncate(1)` command, and editors that `ftruncate` on save either fail or leave stale trailing bytes (write-at-offset-0 overwrites the head but does not shorten the file). Verify the exact failure on the live mount, then implement `setattr`: honor `FATTR_SIZE` by truncating the hydrated cache file and threading the new length through the write-through/`hydration.open_write` commit path; accept chmod/utimes as best-effort/no-op (cloud has no POSIX mode). Acceptance: `printf abc > f; printf '' > f` leaves size 0; `truncate -s N f` works; an editor save that shrinks a file leaves no trailing bytes. `mount/src/fuse_fs.rs` + the write-through path. |

## Medium — efficiency

| Title | Scope |
|---|---|
| fuse3 crate version pin needs upgrade for FUSE_PASSTHROUGH support | `fuse3 = "0.9"` does not expose `FUSE_PASSTHROUGH`/`backing_id`/`setup_passthrough` (grepped the entire crate source — no matches). The Phase-2 read-path therefore falls back to userspace cache reads on hydrated files: `open` still does an IPC `hydration.open_read` to fetch the cache path, but subsequent `read(2)` calls go through the FUSE userspace `read` handler doing `pread` on the cache FD rather than the kernel-direct passthrough path. Correctness unaffected; cost is one extra IPC round-trip per `open` on hydrated files plus userspace round-trips on every read. Re-evaluate when fuse3 ships passthrough (track upstream `Sherlock-Holo/fuse3` for the API), or vendor the kernel ioctl directly via `libc::ioctl(/dev/fuse, FUSE_DEV_IOC_BACKING_OPEN, ...)`. `mount/Cargo.toml`, `mount/src/fuse_fs.rs`. |
| Move `tempfile` from `[dependencies]` to `[dev-dependencies]` | Task 1 (foundation) placed `tempfile` under `[dependencies]` because `fake_jvm.rs` lives in `mount/src/` (gated on `cfg(any(test, debug_assertions))`) and integration tests import it via the library crate. Cargo resolves all `[dependencies]` at release time regardless of cfg gates — so `tempfile` and its transitive deps compile on every release build. Cleaner: move `fake_jvm.rs` into `mount/tests/support/mod.rs` (shared integration-test helper module), declare `tempfile` in `[dev-dependencies]`. Same test coverage, smaller release closure. Today's release binary is 446 KB — not user-visible, but the principle is worth fixing once. `mount/Cargo.toml`, `mount/src/fake_jvm.rs` → `mount/tests/support/mod.rs`. |
| File-manager preview/thumbnail generation triggers bulk hydration of cloud-only files | Browsing the mount in a desktop file manager with previews enabled (live: KDE Dolphin on a mounted Internxt profile) reads file *content* to build thumbnails, and on this cloud-backed FUSE mount every `read(2)` hydrates (downloads) the file via `hydration.open_read` + cache fill. So opening a folder view silently bulk-downloads whatever is in view — defeating the sparse/on-demand model and spiking CPU/network/JVM load. Root cause is structural: FUSE has no online-only-placeholder semantics (unlike the Windows CloudFiles API), so a thumbnailer's read cannot be served without fetching the bytes. Mitigations, cheapest first: (a) signal the mount as a remote/slow filesystem so KDE/GNOME thumbnailers skip auto-previews by default — KDE gates previews on `KFileItem::isSlow()` / `KFileSystemType` detection, GNOME on gvfs/`G_FILE_ATTRIBUTE_FILESYSTEM_*` heuristics; investigate what mount-time signal (fstype string, mount option, or a marker file) flips those. (b) Document advising users to disable previews for the mount as an interim. (c) The full answer is the deferred VFS-placeholder / online-only model (cross-ref the `../unidrive` deferred "Virtual filesystem layer" item) so a stat/preview never hydrates; the planned Dolphin-extension crate could expose explicit pin/unpin instead of implicit hydrate-on-preview. High user impact (makes a file-manager visit expensive), but mitigation (a) is cheap. `mount/src/fuse_fs.rs` (read path) + mount-time fstype/option signalling. |
| `statfs` is unimplemented → `stat -f` / `df` / file managers get ENOSYS | The co-daemon implements no FUSE `statfs` op, so `stat -f <mount>` returns "Function not implemented" (ENOSYS) and `df` can't report the mount. File managers query filesystem free space when entering a folder; ENOSYS there is a known cause of desktop-integration glitches. Fix: implement `statfs` in `mount/src/fuse_fs.rs` returning sane values — `bsize=4096`, `namelen=255`, and `blocks`/`bfree`/`bavail` reflecting the provider quota if cheaply available (else a large constant mostly-free). A static reply needs no IPC; a quota-backed reply could reuse the JVM `quota` info if a cheap verb exists. Acceptance: `stat -f <mount>` succeeds and `df <mount>` shows a row. `mount/src/fuse_fs.rs`. |

## Low — guards and UX

| Title | Scope |
|---|---|
| Cache-file eviction on `unlink`/`rmdir` of hydrated paths | Spec `../unidrive/docs/dev/specs/hydration-namespace-verbs-design.md` §4 R5 / NG4 explicit deferral. When `unlink`/`rmdir` succeed against a hydrated file, the cache copy at `~/.cache/unidrive/hydration/<provider>/<path>` is orphaned — the cloud copy is gone, the state.db row is tombstoned, but the cache file stays. Disk-space concern, not correctness. The existing cache-eviction logic on the JVM side will reap it eventually, but until then it's wasted bytes. Scope: trivial cleanup-on-unlink in `fuse_fs.rs::unlink`/`rmdir` after IPC success — `std::fs::remove_file(cache_path)` (file) or `std::fs::remove_dir_all(cache_dir)` (folder), ignoring NotFound so a never-hydrated path is harmless. The path-to-cache mapping is `<cache_root>/<provider>/<rel>`; `<cache_root>` is the `--cache` flag, `<provider>` is the profile providerId, `<rel>` is the FUSE path. `mount/src/fuse_fs.rs` (unlink/rmdir bodies). |
| `mkdir` parent-missing maps to EIO instead of POSIX ENOENT | Spec `../unidrive/docs/dev/specs/hydration-namespace-verbs-design.md` §4 R3 explicit deferral. `provider.createFolder("/a/b/c")` fails if `/a/b` doesn't exist (OneDrive 404, Internxt similar); the JVM emits a raw provider message and `namespace_err_to_errno` (`mount/src/fuse_fs.rs:172-179`) falls through to the catch-all `EIO`. POSIX says `mkdir` on a non-existent parent returns `ENOENT`. Userland `mkdir -p` creates intermediates one at a time so the kernel always sees a known parent, but a single-call `mkdir` from a program that walks the path itself hits the divergence. Fix shape per spec: JVM-side adds a distinct `parent_not_found` wire-error in `HydrationIpcHandler`; co-daemon adds a `"parent_not_found" => ENOENT` arm to `namespace_err_to_errno`. Both halves coordinate as a single cross-repo change. `mount/src/fuse_fs.rs:172-179` + sibling-repo `HydrationIpcHandler.kt` `hydration.mkdir` branch. |
| Extract `replies()` test helper to `fake_jvm.rs` | The `fn replies(pairs: &[(&str, &str)]) -> HashMap<String, String> { pairs.iter().map(\|(k, v)\| (k.to_string(), v.to_string())).collect() }` body is duplicated byte-for-byte across at least seven sites: `mount/tests/{namespace_ops,getattr_readdir,crash_recovery_replay,write_through,cold_read,ipc_reconnect,ipc_client}.rs` plus `mount/src/cache_scanner.rs`. Each new integration test adds an eighth. Move to `mount/src/fake_jvm.rs` as `pub fn replies(...)`; delete the per-file copies. Pure DRY cleanup, no behavioural change. |
| Collapse the now-10 `ReconnectingIpcClient` io-retry wrappers via a macro | The comment at `mount/src/reconnect.rs:75-77` justifies "three similar copies beat one premature `dyn Future` indirection." That was true at 3. The count is now 10 (`open_read`, `open_write`, `close_handle`, `hydrate`, `dehydrate`, `last_synced`, `list`, `mkdir`, `unlink`, `rmdir`) — every method is `loop { ensure_connected; match c.<verb>(args).await { Ok(v) => return Ok(v), Err(Io(_)) => { self.inner = None; continue; }, Err(e) => return Err(e) } }`. A `macro_rules! retry_io_loop` would collapse ~120 lines to ~20 with no `dyn Future` (the original concern). The lifetime objection in the comment doesn't apply to macro expansion. Each new verb the JVM adds (Phase 3 events, future namespace ops) currently means another 10-line copy-paste. Acceptance: macro + 10 single-line invocations, update the load-bearing comment to point at the macro definition. `mount/src/reconnect.rs`. |
| Path-construction idiom drift between `lookup` and namespace ops | `lookup` (`fuse_fs.rs:240-244`) uses `if parent_path.is_empty() { format!("/{name}") } else { format!("{parent_path}/{name}") }`. The three namespace ops (mkdir/unlink/rmdir at `:652`, `:693`, `:728`) use `format!("{}/{}", parent_path.trim_end_matches('/'), name)`. Both produce identical strings for the cases the codebase exercises (root: `/foo`; nested: `/folder/foo`). Pure stylistic divergence; future readers will burn cycles wondering whether the two idioms encode different invariants. Acceptance: extract `fn child_path(parent: &str, name: &str) -> String` near `basename` (`:154-160`), call from all four sites; tests unchanged. `mount/src/fuse_fs.rs`. |
| Directories always report `nlink=2` regardless of subdirectory count (leaf-directory optimization gap) | `file_attr_from_cached` sets `nlink: if c.is_folder { 2 } else { 1 }` (`mount/src/fuse_fs.rs` ~219) for every directory. POSIX directories have `nlink = 2 + (immediate-subdirectory count)`; tools that use the "leaf optimization" (`find` without `-noleaf`, `tar`, some indexers/backup tools) read `nlink==2` as "no subdirectories" and skip descending — so they'd miss subfolders on the mount. `ls` and most GUI file managers don't rely on it, so impact is narrow. Computing the true count at `getattr` time needs a child listing (IPC round-trip), which is costly; options: (a) accept `nlink=2` and document the `find -noleaf` caveat, (b) fill it opportunistically from a warm `populate_from_list` result, or (c) report a benign sentinel `>2`. Low priority — narrow tool subset. `mount/src/fuse_fs.rs`. |
| xattr ops (`getxattr`/`listxattr`/`setxattr`/`removexattr`) unimplemented | The co-daemon implements no xattr ops, so they fall to the fuse3 default (ENOSYS → kernel surfaces ENOTSUP). Desktop stacks query xattrs/ACLs (`system.posix_acl_access`, `security.selinux`, KDE/GNOME metadata); ENOTSUP is usually tolerated, but some tools log errors or skip files. Lower-stakes than `setattr`. Scope: implement `listxattr` returning empty and `getxattr` returning `ENODATA`/`ENOTSUP` cleanly (explicitly, not a generic EIO); reject `setxattr` with ENOTSUP (cloud has no xattr store) unless a use surfaces. Acceptance: `getfattr -d <path>` returns cleanly, file managers log no xattr failures. `mount/src/fuse_fs.rs`. |
| No way to see hydration state (local vs cloud-only) from within the mount | The mount renders cloud-only and hydrated files identically — a user browsing can't tell which files are physically present vs cloud-only (will download on read). The data already exists: `hydration.list` carries `isHydrated` and `getattr` holds it (`CachedAttr.is_hydrated`); it just isn't surfaced. Options: (a) a synthetic xattr (`user.unidrive.hydrated`) once xattr ops land, (b) a Dolphin-extension overlay icon (the planned extension crate) reading the flag, (c) leave it to `unidrive ls`/`status` out-of-band. Pure UX-transparency nicety. `mount/src/fuse_fs.rs` (`get_or_fetch_attr` already carries `is_hydrated`) + the planned Dolphin extension. |
| Optimise truncate-to-`N`>0 of a cloud-only file (prefix/Range download, not whole-file) | Per `../unidrive/docs/dev/specs/setattr-truncate-design.md`: `setattr(size=N)` for N>0 on a non-hydrated file currently `open_read`-downloads the **whole** file before `set_len(N)` — even when shrinking, the discarded tail is fetched for nothing. Optimisation: fetch only the first N bytes via a Range request (where the provider supports it — cf. the unidrive "OneDrive Range against `@microsoft.graph.downloadUrl`" item) so the cache holds exactly the kept prefix. Scope notes: already-hydrated files are unaffected (`open_read`/`ensureHydrated` returns the warm cache — no re-download); a pure **grow** of a cloud-only file still needs the existing bytes to preserve them, so the spec's "grow can skip download" note holds only for already-hydrated files. Low — truncate-to-N>0 is rare; truncate-to-0 (the common `>` case) already avoids download by design. Cross-repo: depends on a provider Range-download capability. |

## Out of scope across all surfaces

- Windows or macOS support in this binary. The Windows Cloud Files API placeholder surface lives in a separate platform tier per `../unidrive/docs/adr/multi-platform.md`.
- Auth, sync, provider logic, OAuth flows. Those belong in the JVM daemon.
- KIO slave implementation. Skipped at first; promote from optional only if Dolphin ServiceMenus prove insufficient.
- Graceful degrade to kernels < 6.9. The kernel-floor commitment is the design.
- Auto-restart of a crashed co-daemon. Explicit user re-mount only.
- Two concurrent `unidrive mount` invocations on the same path. Relies on kernel-level rejection of the second `fusermount3`.
- Running as root. Refused with an explicit error; mount is per-user.
