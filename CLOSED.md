- **Co-daemon-side `flock(2)` of profile lock file (mode-mutex R4 option (b))** — New optional `--lock <path>` CLI flag; when supplied, the co-daemon opens the per-profile `.lock` file (creating it if missing, matching JVM `ProcessLock` open semantics) and acquires `flock(LOCK_EX | LOCK_NB)` after the IPC connect but before `mount_with_unprivileged`. The FD lives for the FUSE session lifetime so the kernel-on-exit release covers both orderly shutdown (Drop) and SIGKILL. On contention the binary refuses with exit 1 and stderr `"Profile lock <path> is held by another unidrive process; refusing mount."` — matches the spec §4 R4 option (b) close of the `kill -9 <jvm-pid>` race that left the JVM-side `ProcessLock` released but the co-daemon still serving the FUSE mount. New `mount/src/profile_lock.rs` with `ProfileLock::acquire()` wrapping `libc::flock`; new integration tests `mount/tests/profile_lock.rs` cover (a) the binary holds the flock for the session lifetime, asserted by an independent in-process `flock(LOCK_EX | LOCK_NB)` returning EWOULDBLOCK while the binary runs, and (b) the binary refuses startup with exit 1 + "refusing mount" stderr when the test pre-acquires the lock. Sibling-repo follow-up filed for JVM-side `MountCommand.kt` to pass `--lock <path>` matching the per-profile config-dir layout the spec defines; until that lands, mounts launched without `--lock` preserve today's behaviour. Spec: `../unidrive/docs/dev/specs/mount-sync-mode-mutex-design.md` §4 R4.
- **lookup O(n) attrs-scan collapsed to O(1) via PathMap::inode_for** — `fuse_fs.rs::lookup` no longer walks every cached attr entry comparing each inode's path to the target. `PathMap::inode_for` (added by the namespace-verbs audit) gives the reverse lookup directly. - **Cache-file eviction on unlink/rmdir of hydrated paths** — Spec `../unidrive/docs/dev/specs/hydration-namespace-verbs-design.md` §4 R5 / NG4. `UnidriveFs` gains an optional `cache_root: Option<PathBuf>` field wired via a new `with_cache_root(PathBuf) -> Self` builder method. `run.rs` plumbs the `--cache <path>` flag (already parsed for `scan_and_replay`) into `UnidriveFs::new(...).with_cache_root(cache_root.to_path_buf())`. After a successful `hydration.unlink` IPC, `unlink` calls `std::fs::remove_file(<cache_root>/<rel>)`; `rmdir` calls `std::fs::remove_dir_all(<cache_root>/<rel>)`. NotFound is silent (never-hydrated paths are harmless); any other I/O error logs at warn and the user's `rm` still succeeds (cloud + state.db are already gone). Three new integration tests in `mount/tests/cache_eviction.rs`: `unlink_removes_hydrated_cache_file`, `unlink_tolerates_missing_cache_file`, `rmdir_removes_hydrated_cache_directory`.
