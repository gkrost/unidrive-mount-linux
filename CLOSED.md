- **Write durability: `fsync` commits to the cloud + scanner reliably replays create-then-crash** — Closes the High-tier entry where `fsync` gave zero cloud-durability guarantee and a just-created-then-crashed file could slip past the recovery scanner. Co-daemon half (this repo): `UnidriveFs::fsync` now, after flushing the cache FD, fires `hydration.open_write(handle_id, remote_path, cache_path)` and AWAITS it when the handle is dirty. On success it clears the per-handle `dirty` flag so the subsequent RELEASE doesn't re-upload identical bytes; on failure it maps the `IpcError` through `ipc_error_to_errno` and returns the errno to userland — the key behavioural difference from RELEASE, where `close(2)` has already returned and the kernel ignores release error codes, so an app that explicitly `fsync`s for durability now learns when the cloud commit failed instead of getting a silent success. The handle's identity (`handle_id`/`remote_path`/`cache_path`/`dirty`) is cloned out from under the `open_handles` lock before the IPC await, mirroring RELEASE's lock discipline; dirty is re-fetched under the lock after the await since the fh may have been released concurrently. RELEASE's `close_handle` was already unconditional (fired outside the `if dirty` block), so it correctly still fires after a dirty-clearing fsync to release the JVM open-set entry — no un-gating was needed. Write ordering is unchanged: a `write` after a successful fsync re-sets dirty, so write-fsync-write-close still uploads the final bytes at RELEASE. `cache_scanner::scan_and_replay` tightened from strict `mtime > last_synced` to `mtime >= last_synced`: `create` stamps `last_synced = now`, so a created-then-crashed file has cache mtime equal to (or rounding just below) the watermark and a strict `>` lets it slip through and never reach the cloud. Replay-on-tie is the conservative choice — a spurious re-upload of an already-synced file is idempotent (`open_write` re-puts identical bytes) and cheap, whereas a missed upload is silent cloud-durability loss; the existing watermark-skip tests use a watermark of 1ms so they are unaffected. The companion JVM-side work (failure-surfacing on the upload path plus marking the state.db row unsynced so an operator/`doctor` check can observe a deferred upload) lands in parallel in the sibling repo. New integration tests in `mount/tests/fsync_durability.rs`: `fsync_fires_open_write_and_awaits` (open_write observed before fsync(2) returns; the post-fsync RELEASE fires exactly one — no second — open_write but still fires close_handle) and `fsync_returns_eio_when_upload_fails` (JVM rejects the upload; the `fsync(2)` syscall returns an error to userland rather than silently succeeding).
